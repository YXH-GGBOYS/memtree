# /memtree_bootstrap — Build .memory/ for Your Project

## Usage
```
/memtree_bootstrap                # Full build from memtree.config.yaml
/memtree_bootstrap --resume       # Resume from last checkpoint (if interrupted)
```

## Prerequisites
- `memtree.config.yaml` exists in your project root (generated by `/memtree_init`)
- MemTree repository is cloned locally (prompts and scripts are referenced from the clone)
- Set `MEMTREE_HOME` env var to point to your MemTree clone: `export MEMTREE_HOME=~/memtree`
- Or run all commands from within the MemTree clone directory
- Read the config file first to understand the project structure

## Execution Steps

### Phase 1: Scan Code Chains
1. Read `memtree.config.yaml`
2. For each service, Glob scan all source files (respecting `exclude` patterns)
3. For each entry point file, Read it and extract import statements
4. Recursively follow imports (project-internal only, stop at third-party)
5. Group into chains: one entry + all its recursive dependencies = one chain
6. Identify shared files: imported by >= `advanced.shared_threshold` chains
7. Output: `.memory/chains.json` + `.memory/shared_files.json`

### Phase 2a: Analyze Shared Files
1. Read `.memory/shared_files.json`
2. For each shared file, use the Worker Shared prompt (`prompts/build/worker-shared.md`):
   - Read the actual source file
   - Extract function signatures, imports, exports
   - Document constraints (transaction mode, currency units, etc.)
   - Record "multi-chain impact" — which business features are affected
3. Write `.memory/{service}/{path}.md` for each shared file
4. Checkpoint: `echo "SHARED" >> .memory/.checkpoint`

### Phase 2b: Analyze DB Schema (if database configured)
1. Read `database` section from config
2. Execute SQL queries via the configured access command:
   - List all schemas and tables
   - For key tables: column definitions, constraints, foreign keys
3. For each table, grep ORM model files to find mapping mismatches
4. Write `.memory/db/{schema}/{table}.md` for each table
5. Checkpoint: `echo "DB" >> .memory/.checkpoint`

### Phase 2c: Analyze Regular Chains
1. Read `.memory/chains.json`, skip shared files (already done)
2. Process in batches of `advanced.batch_size`:
   - For each chain, use the Worker Analyze prompt (`prompts/build/worker-analyze.md`)
   - Read each file in the chain
   - Generate per-file memory with: frontmatter + TL;DR + Quick Ref + Full Analysis
3. After each batch, append completed chain IDs to `.memory/.checkpoint`
4. If `--resume`: read `.checkpoint` and skip completed chains

Note: Due to context limits, a single bootstrap run may not deep-analyze every file.
The skeleton generator creates TL;DR entries for all files. For files needing deeper analysis,
use `/memtree_rebuild <service>` after bootstrap to enrich specific services.
This is by design — skeleton + TL;DR provides 80% of the value, deep analysis is incremental.

### Phase 3: Parse Worker Output
- Parse all worker output and write individual `.memory/{service}/{path}.md` files
- Ensure frontmatter includes: source, service, layer, source_hash, depends_on, depended_by

### Phase 4: Generate Indexes
1. **INDEX.md** per directory: list all files with their TL;DR
2. **PITFALLS.md** per service:
   - Import from `memtree.config.yaml` pitfalls section (team knowledge)
   - Add pitfalls discovered during Worker analysis
   - Categorize: 🔴 Critical (data corruption/500) vs 🟡 Warning (logic bugs)
3. **cross-refs/** directory:
   - `orm-db-mismatch.md`: ORM attribute name ≠ DB column name
   - `field-confusion.md`: same name, different meaning across services
   - `api-field-mapping.md`: frontend → API → DB column
   - `service-schema-matrix.md`: which service reads/writes which schema
   - `INDEX.md`: navigation guide
4. **ROOT.md**: project overview + navigation + freshness info

### Phase 5: Validate Structure
Run validation checks:
- Every `depends_on` entry has a matching `depended_by` in the target file
- Every `source_hash` matches the current source file
- No orphaned .memory/ files (source file deleted)
- Every directory has an INDEX.md
- Every service has a PITFALLS.md

### Phase 6: Quality Audit
Sample `quality.sample_count` files (default 15) and verify:
- TL;DR accurately describes the file's role
- Quick Ref function signatures match actual source code
- depends_on covers all project-internal imports
- depended_by includes at least 1 known caller
- source_hash is correct
- PITFALLS.md pointers are valid

Report: X/15 PASS. If >= `quality.pass_threshold`: success. Otherwise: list failures and suggest prompt improvements.

### Completion
- Remove `.memory/.checkpoint` (full build complete)
- Show the user a summary of generated files
- Ask: "Ready to commit .memory/ to git? [Y/n]"
- Only commit if user confirms
- Report summary: total files, PASS rate, any issues found

## Error Handling
- If a Worker fails on a specific chain → log the error, continue with next chain
- If SSH/DB access fails → skip DB phase, warn user, continue with code-only analysis
- If quality audit fails → DON'T delete .memory/, report issues so user can fix prompts and re-run specific services with `/memtree_rebuild`

## Interruption Recovery
`/memtree_bootstrap --resume` reads `.memory/.checkpoint` and skips completed phases/chains.
If no checkpoint exists, starts from scratch.
